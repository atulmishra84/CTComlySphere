# CT ComplySphere — Azure Deployment Guide

Step-by-step instructions for deploying CT ComplySphere to Azure Container Apps.

---

## Prerequisites

Install the following tools on your local machine before you begin.

1. **Azure CLI** — <https://learn.microsoft.com/en-us/cli/azure/install-azure-cli>
   ```bash
   az version   # should print 2.50 or later
   ```

2. **Docker Desktop** (or Docker Engine) — <https://docs.docker.com/get-docker/>
   ```bash
   docker --version
   ```

3. Log in to Azure:
   ```bash
   az login
   az account set --subscription "<YOUR_SUBSCRIPTION_ID>"
   ```

---

## Step 1 — Copy and fill in environment variables

```bash
cp .env.example .env
```

Open `.env` in your editor and set at minimum:

| Variable | Description |
|---|---|
| `DATABASE_URL` | PostgreSQL connection string (auto-generated by the script — leave blank if using `azure-deploy.sh`) |
| `SESSION_SECRET` | Random secret for Flask session cookies |

The cloud-scanning variables (`AWS_*`, `AZURE_*`, `GCP_*`) are optional; the app starts without them.

> **Note:** `azure-deploy.sh` auto-generates `DATABASE_URL` and `SESSION_SECRET` from scratch during deployment, so you only need to fill in `.env` for local development or manual deployments.

---

## Step 2 — Run the deployment script

```bash
chmod +x azure-deploy.sh
./azure-deploy.sh
```

The script performs nine steps automatically:

1. Verifies your Azure CLI session
2. Creates a resource group (`complysphere-rg` in `eastus`)
3. Creates an Azure Container Registry (`complysphereacr`)
4. Builds and pushes the Docker image via `az acr build`
5. Provisions an Azure Database for PostgreSQL Flexible Server
6. Creates a Container Apps managed environment
7. Deploys the container app with `DATABASE_URL` and `SESSION_SECRET` injected as environment variables
8. Runs a one-off seed job (`seed_azure_data.py`) to populate initial data
9. Prints the live URL, database details, and the generated secrets

> **Save the output.** The script prints the generated database password and session secret — store them securely (e.g., in Azure Key Vault or a password manager).

---

## Step 3 — Verify the deployment

Once the script finishes it prints the app URL. Confirm the app is running by hitting the health endpoint:

```bash
APP_URL="<paste the URL printed by the script>"
curl -sf "https://${APP_URL}/health"
```

Expected response (HTTP 200):

```json
{
  "status": "ok",
  "timestamp": "2026-04-03T10:00:00Z",
  "version": "1.0.0"
}
```

If `curl` returns an error, check the container logs:

```bash
az containerapp logs show \
  --name complysphere-app \
  --resource-group complysphere-rg \
  --follow
```

Azure Container Apps uses `/health` as the liveness and readiness probe by default once the endpoint is available.

---

## Step 4 — Open the app

Navigate to `https://<APP_URL>` in your browser. The CT ComplySphere dashboard should load.

---

## Teardown

To delete every Azure resource created by the deployment (including the database and all data):

```bash
az group delete --name complysphere-rg --yes --no-wait
```

This removes the resource group and everything inside it asynchronously.

---

## Troubleshooting

| Symptom | Likely cause | Fix |
|---|---|---|
| Container exits immediately on start | Missing `DATABASE_URL` or `SESSION_SECRET` | Check the container app environment variables in the Azure Portal |
| `/health` returns 500 | Database unreachable | Verify firewall rules on the PostgreSQL Flexible Server allow the Container Apps outbound IP |
| Seed job fails | Database not yet ready | Wait 2–3 minutes and re-trigger: `az containerapp job start --name complysphere-seed --resource-group complysphere-rg` |
| `az acr build` fails | Docker build error | Run `docker build .` locally to see the full error |
