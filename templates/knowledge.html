What Is CT ComplySphere?

CT ComplySphere Visibility & Governance Platform is an enterprise-grade, AI-agent discovery, compliance, and governance system built for healthcare and regulated industries. It automatically discovers, classifies, scans, and evaluates every AI system operating in your environment — visible or hidden — against major regulatory frameworks.

Whether your AI agents run in Kubernetes clusters, Docker containers, REST APIs, robotic systems (Clawbots), or shadow deployments your team doesn't know about, CT ComplySphere finds them, assesses their risk, and provides step-by-step remediation playbooks to achieve compliance.

Compliance frameworks covered: HIPAA · HITRUST CSF · FDA SaMD · GDPR · SOC 2 Type II

Platform Components & Architecture
Discovery Engine

Continuously scans your network and infrastructure to detect all AI agents.

  • Kubernetes pod & service discovery
  • Docker container scanning
  • REST / gRPC / WebSocket / GraphQL probing
  • MQTT broker agent detection
  • MCP (Model Context Protocol) scanner
  • Cloud service enumeration (AWS/Azure/GCP)
Compliance Engine

Evaluates every discovered agent against regulatory frameworks with scored reports.

  • HIPAA Technical & Administrative Safeguards
  • HITRUST CSF Control Mapping
  • FDA SaMD (Software as Medical Device)
  • GDPR Article 22 / Data Processing
  • SOC 2 Type II Trust Criteria
  • Configurable scoring thresholds per org
Shadow AI Detection

Identifies unauthorized, rogue, or unregistered AI deployments that bypass governance.

  • Behavioral anomaly fingerprinting
  • Network traffic pattern analysis
  • Unregistered model endpoint detection
  • PHI exposure identification
  • Risk-ranked shadow agent inventory
  • One-click remediation escalation
Clawbot / Robotic AI Module

Dedicated detection and registration for physical robotic AI systems in healthcare.

  • ROS (Robot Operating System) network scan
  • MQTT telemetry broker detection
  • VEX Robotics controller discovery
  • Safety system validation (SIL, collision, force)
  • PHI access risk scoring for robots
  • FDA SaMD compliance check for surgical robots
Analytics & Risk Engine

Predictive scoring and trend analysis for proactive risk management.

  • Weighted vulnerability risk scoring
  • Trend forecasting (30/60/90-day windows)
  • PHI exposure heatmaps
  • Compliance score drift alerts
  • Protocol vulnerability comparison
  • Executive summary dashboards
Remediation Playbooks

Structured, step-by-step remediation guides mapped to each compliance finding.

  • Framework-specific playbook library
  • Automated playbook generation from scan findings
  • Assignable tasks with due dates
  • Playbook versioning and duplication
  • Progress tracking and audit trail
  • Integration with ticketing systems (roadmap)
Collector Agent System

Deployable lightweight agents that send telemetry from remote or air-gapped environments.

  • Python-based collector package (pip install)
  • Token-authenticated secure registration
  • Periodic heartbeat and agent reporting
  • Environment labeling (prod/staging/dev)
  • Works behind corporate firewalls / NAT
  • Auto-revokable via platform console
Multi-Cloud Manager

Unified view of AI deployments across all major cloud providers.

  • AWS: SageMaker, Bedrock, Lambda, EKS
  • Azure: ML Studio, AKS, Container Instances
  • GCP: Vertex AI, GKE, Cloud Run
  • Cross-cloud compliance gap analysis
  • Cloud-native IAM policy review
  • Cost & risk optimization recommendations
Audit & Governance Trail

Immutable audit logging for every compliance action and security event.

  • Tamper-evident audit log records
  • User action tracking (who changed what)
  • Compliance evaluation history
  • Scan result diffs over time

On-Premises Deployment

Ideal for hospitals and enterprises with strict data residency requirements, air-gapped networks, or internal security policies that prohibit cloud egress of PHI metadata.

Recommended for healthcare organizations with PHI data and regulatory data-residency mandates.
Deployment Steps
  1. Provision a Linux server (Ubuntu 22.04 LTS recommended, RHEL 8+ supported)
  2. Install Docker Engine + Docker Compose, or a Kubernetes distribution (k3s / RKE2)
  3. Pull the CT ComplySphere platform image from your private registry
  4. Configure DATABASE_URL to point to your internal PostgreSQL instance
  5. Set SESSION_SECRET and all required environment variables
  6. Configure reverse proxy (Nginx / HAProxy) with TLS termination
  7. Open internal network ports (see network prerequisites)
  8. Deploy collector agents on target subnets for distributed scanning
  9. Run database migrations: flask db upgrade
  10. Start the platform: gunicorn main:app --bind 0.0.0.0:5000

AWS Deployment

Deploy CT ComplySphere on AWS using ECS Fargate (recommended) or EC2 with EKS. Use RDS PostgreSQL for managed database, ALB for load balancing, and ACM for TLS certificates.

Recommended Architecture
  • Compute: ECS Fargate (serverless containers) or EC2 t3.large+
  • Database: Amazon RDS PostgreSQL (Multi-AZ for HA)
  • Load Balancer: Application Load Balancer (ALB) with HTTPS listener
  • TLS: AWS Certificate Manager (ACM) — free managed certs
  • Secrets: AWS Secrets Manager for SESSION_SECRET, DATABASE_URL
  • Container Registry: Amazon ECR for platform image
  • Monitoring: CloudWatch Logs + Container Insights
  • Network: VPC with private subnets for app/db, public for ALB
  • Scanning: Deploy collector agents on EC2 worker nodes via SSM
AWS IAM Permissions Required

For ECS Task Role:

  • secretsmanager:GetSecretValue
  • rds-db:connect
  • logs:CreateLogStream, logs:PutLogEvents

For AI Discovery (Optional):

  • eks:DescribeCluster, eks:ListClusters
  • sagemaker:ListEndpoints, sagemaker:DescribeEndpoint
  • lambda:ListFunctions
  • ec2:DescribeInstances, ec2:DescribeSecurityGroups
  • bedrock:ListFoundationModels
Use HIPAA-eligible services only. Ensure your AWS account has a signed BAA with Amazon before storing PHI metadata.
Azure Deployment

Deploy on Azure using Azure Container Apps (recommended) or AKS. Use Azure Database for PostgreSQL, Azure Application Gateway for TLS, and Key Vault for secrets management.

Recommended Architecture
  • Compute: Azure Container Apps or AKS (Standard D2s_v3+)
  • Database: Azure Database for PostgreSQL — Flexible Server
  • Load Balancer: Azure Application Gateway with WAF v2
  • TLS: Azure-managed certificates or Let's Encrypt via cert-manager
  • Secrets: Azure Key Vault with Managed Identity access
  • Container Registry: Azure Container Registry (ACR)
  • Monitoring: Azure Monitor + Log Analytics Workspace
  • Network: VNet with private endpoints for DB and Key Vault
  • Scanning: Collector agents via Azure Arc for hybrid environments
Azure RBAC Roles Required

Platform Service Principal:

  • Key Vault Secrets User — read secrets
  • AcrPull — pull container images
  • Monitoring Metrics Publisher — emit metrics

For AI Discovery (Optional):

  • Reader on ML Workspace
  • Azure Kubernetes Service Cluster User
  • Container Instance Contributor (read-only)
  • Cognitive Services Reader
Use Azure Government Cloud (Azure Gov) for US federal healthcare customers to ensure FedRAMP / HIPAA compliance.
Google Cloud Deployment

Deploy on GCP using Cloud Run (recommended for simplicity) or GKE Autopilot. Use Cloud SQL for PostgreSQL, Cloud Load Balancing for TLS, and Secret Manager for credentials.

Recommended Architecture
  • Compute: Cloud Run (fully managed) or GKE Autopilot
  • Database: Cloud SQL for PostgreSQL (HA with failover replica)
  • Load Balancer: Google Cloud Load Balancing with managed SSL
  • TLS: Google-managed SSL certificates
  • Secrets: Secret Manager with Workload Identity
  • Container Registry: Artifact Registry
  • Monitoring: Cloud Monitoring + Cloud Logging
  • Network: VPC with Private Google Access enabled
  • Scanning: Collector agents via Cloud Run Jobs or GCE
GCP IAM Roles Required

Platform Service Account:

  • roles/secretmanager.secretAccessor
  • roles/cloudsql.client
  • roles/logging.logWriter
  • roles/monitoring.metricWriter

For AI Discovery (Optional):

  • roles/aiplatform.viewer — Vertex AI
  • roles/container.clusterViewer — GKE
  • roles/run.viewer — Cloud Run services
  • roles/compute.viewer — GCE instances
Use VPC Service Controls to restrict data perimeter and meet HIPAA requirements on GCP.
Multi-Cloud Deployment

For organizations running AI workloads across two or more cloud providers simultaneously. CT ComplySphere acts as the central governance plane with collector agents deployed per-cloud.

Reference Architecture
  1. Central Platform Instance — Deploy the CT ComplySphere core on one primary cloud or on-premises. This is the authoritative governance hub.
  2. Per-Cloud Collector Agents — Deploy a lightweight collector agent in each cloud environment (AWS VPC, Azure VNet, GCP VPC). Agents register with the central platform via HTTPS and send telemetry securely.
  3. Cross-Cloud Network — Use Transit Gateway (AWS), VNet Peering (Azure), or Cloud VPN to enable agents to reach the central instance, or use the public internet with TLS + token auth.
  4. Unified Database — All findings land in a single PostgreSQL instance (managed or self-hosted). Use multi-region replicas for resilience.
  5. Federated IAM — Each cloud uses its own service account / IAM role with least-privilege read access for AI service discovery.

Prerequisites Before Deployment
Software & Runtime
  • Python 3.10+ (3.11 recommended)
  • PostgreSQL 14+
  • Docker Engine 24+ (optional for container scanning)
  • Kubernetes 1.26+ (optional for K8s scanning)
  • Nginx or HAProxy for reverse proxy + TLS
  • gunicorn 21+ for production serving
  • pip packages: Flask, SQLAlchemy, psycopg2, etc. (see requirements.txt)
  • OpenSSL for TLS certificate generation
Network & Infrastructure
  • Network read-access to target subnets hosting AI agents
  • DNS resolution for target hostnames
  • Outbound HTTPS (port 443) for cloud API calls
  • Static IP or DNS hostname for platform URL
  • TLS certificate (self-signed for internal, CA-signed for external)
  • Firewall rules allowing probe traffic to AI agent ports
  • NTP time sync across all nodes (critical for audit timestamps)
Organizational Requirements
  • Designated Platform Administrator account
  • Inventory list of known AI systems / endpoints (optional but helpful)
  • Cloud provider service account credentials (for cloud scanning)
  • Kubernetes kubeconfig with read permissions (for K8s scanning)
  • Docker daemon socket access (for Docker scanning)
  • MQTT broker credentials (for Clawbot scanning)
  • ROS Master URI (for robotic system discovery)
  • Security team approval for network scan scope
Healthcare environments: Ensure all scanning activity is approved by your HIPAA Security Officer and is scoped to avoid triggering IDS/IPS alerts on clinical networks.
